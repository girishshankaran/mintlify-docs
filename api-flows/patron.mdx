# Patron API Flow

This API flow documents how a patron browses dishes, checks whether ingredients are in stock, views dish details, and places an order.

## 1. Start Browsing

### Request

The client sends an HTTP request to retrieve dishes in a specific category so the patron can start browsing the menu.

```bash
curl -X GET "<BASE_URL>/dishes?filter=category.eq~Appetizer" \
  -H "Accept: application/json"
```

### Response

If the request was successful, the server responds with a 200 status code and a response body containing a list of dishes in the selected category. The client can use the `id`, `name`, and `ingredients` fields to decide which dish to inspect next.

```json
{
  "results": [
    {
      "id": "66294b2a4475a41f3e709bd1",
      "created_at": "2024-04-22T10:00:00Z",
      "name": "Risotto alla Milanese",
      "description": "A description of the resource.",
      "category": "Appetizer",
      "image_name": "burger",
      "ingredients": [
        {
          "ingredient_id": "66294b2a4475a41f3e709bc5",
          "is_essential": true
        },
        {
          "ingredient_id": "66294b2a4475a41f3e709bc6",
          "is_essential": true
        }
      ],
      "preparation_time": 999,
      "price": 999,
      "station": "hot"
    }
  ],
  "total_results": 1
}
```

## 2. Check Ingredient Quantities

### Request

The client sends an HTTP request to retrieve the details of a specific ingredient so it can confirm the ingredient has a positive `in_stock_qty`.

```bash
curl -X GET "<BASE_URL>/ingredients/66294b2a4475a41f3e709bc5" \
  -H "Accept: application/json"
```

### Response

If the request was successful, the server responds with a 200 status code and a response body containing the ingredient details.

```json
{
  "id": "674d0bf5c28b69001f8e03a1",
  "created_at": "2024-04-22T10:00:00Z",
  "updated_at": "2024-04-22T10:00:00Z",
  "name": "Carrot",
  "in_stock_qty": 3,
  "price": 999
}
```

## 3. View Dish Details

### Request

The client sends an HTTP request to retrieve the details of a specific dish so the patron can review the full dish information before ordering.

```bash
curl -X GET "<BASE_URL>/dishes/66294b2a4475a41f3e709bd1" \
  -H "Accept: application/json"
```

### Response

If the request was successful, the server responds with a 200 status code and a response body containing the dish details.

  "created_at": "2024-04-22T10:00:00Z",

## 4. Add Dishes and View the Cart

### Request

No curl request is needed for this step. In the user interface, the patron adds one or more dishes to the cart, updates quantities if needed, and reviews the current cart contents before checkout.

### Response

No API response is documented for this step.

## 5. Place an Order

### Request

The client sends an authenticated HTTP request to create an order.

```bash
curl -X POST "<BASE_URL>/orders" \
  -H "Accept: application/json" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -d '{
    "name": "Benjamin",
    "table_number": 18,
    "dish_ids": [
      "66294b2a4475a41f3e709bd1"
    ],
    "special_requests": "No peanuts",
    "scheduled_at": null
  }'
```

### Response

If the request was successful, the server responds with a 201 status code and a response body containing the newly created order.

```json
{
  "id": "674d0bf5c28b69001f8e03a1",
  "created_at": "2024-04-22T10:00:00Z",
  "updated_at": "2024-04-22T10:00:00Z",
  "name": "John",
  "table_number": 8,
  "dish_ids": [
    "66294b2a4475a41f3e709bd1"
  ],
  "special_requests": "No peanuts",
  "priority": 3,
  "scheduled_at": null,
  "status": "Received"
}
```

## 6. Order Confirmation

### Request

No curl request is needed.

### Response

No additional API response beyond the `201 Created` response.
